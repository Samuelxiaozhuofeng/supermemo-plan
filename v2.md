# SuperMemo Plan Core App v2 功能清单

## 目标

v2 的目标不是无限扩张功能，而是在 v1 已有的“模板 + 今日副本 + 比例压缩 + 执行重算 + 延误分析”基础上，把它变成一个可长期使用的个人时间计划工具。

优先方向：

1. 提高 SuperMemo Plan 还原度。
2. 补齐长期复盘能力。
3. 改善执行中的临时调整体验。
4. 增强本地数据安全和可迁移性。
5. 拆分当前前端结构，降低后续维护成本。

---

## P0：v2 最值得优先做的功能

### 1. Adjust：从执行结果反推模板

SuperMemo Plan 的核心反馈闭环是：先按理想值规划，执行后通过延误分析修正模板。

建议加入：

- `Adjust selected template`：把今日副本的实际分配结果写回来源模板。
- `Adjust from today`：按今日每个活动的实际/分配时间更新模板的 `desiredMinutes`。
- 调整前显示预览：
  - 原理想时长
  - 今日实际时长
  - 新模板时长
  - 变化量
- 支持逐项勾选，不强制全部覆盖。
- 对拆分/合并后的活动给出明确策略：
  - 标题完全相同：自动匹配。
  - 标题不匹配：让用户手动映射到模板活动。
  - 无法匹配：作为新活动追加或忽略。

### 2. 历史执行记录

v1 只有一个今日副本，v2 应保存每天的执行结果。

建议加入：

- 历史日程列表。
- 按日期、模板、完成状态筛选。
- 打开历史日程只读查看。
- 从历史日程复制为新的今日副本。
- 历史记录可删除，但需要确认。
- 今日副本 Terminate 后自动进入历史记录。

### 3. 导出

先做本地导出，不做云端。

建议加入：

- 导出今日执行记录为 Markdown。
- 导出今日执行记录为 CSV。
- 导出延误分析为 CSV。
- 导出全部本地数据为 JSON 备份。
- 从 JSON 备份恢复。

Markdown 导出示例：

```md
# 2026-04-30 学习与深度工作

- 08:30 - 启动：整理桌面与计划 (25 分)
- 08:55 - SuperMemo：核心复习 (70 分)
- 10:05 - 深度工作：写作 / 项目 (145 分)

## 延误分析

- SuperMemo：核心复习 +12 分
- 邮件与杂务 -8 分
```

### 4. 更完整的执行编辑

v1 已有新增、删除、排序、拆分、合并。v2 可以补齐几个高价值操作：

- 跳过活动：
  - 将活动标记为 skipped。
  - 不从统计中删除，但单独显示“跳过”。
  - 剩余时间重新分配给后续活动。
- 完成当前活动：
  - 比 Terminate 更轻，只结束当前活动，不结束整个计划。
- 插入紧急活动：
  - 选择时长或结束时间。
  - 插入后自动压缩前后剩余活动。
- 手动修正实际开始/结束时间：
  - 用于忘记点击 Begin 的情况。
  - 修改必须可见，不能静默改历史。

### 5. 改进提醒系统

v1 已有 app 打开时的提醒。v2 可以增强：

- 提前提醒，例如结束前 3 分钟提醒。
- 贪睡时长可选：3 / 5 / 10 / 自定义分钟。
- 当前活动结束后持续显示 overdue 状态。
- 提醒权限状态明确展示：
  - 未请求
  - 已允许
  - 已拒绝
  - 当前浏览器不支持
- 音效开关。
- 静音当前活动。

## P1：SuperMemo Plan 还原增强

### 1. Rigid 刚性时段

Rigid 活动始终保持用户设定的 `desiredMinutes`，不参与比例压缩。

建议行为：

- 活动新增 `rigid: boolean`。
- 刚性活动优先占用时长。
- 剩余活动按比例压缩。
- 如果刚性活动总时长超过计划总时长，直接显示错误状态，不吞掉问题。
- UI 中用锁图标显示刚性时段。

### 2. OptStart / OptLen / OptShift 字段

v1 已有实用近似字段。v2 可以更接近 SuperMemo 原始表格：

- `OptStart`：无固定时间影响时的理论开始时间。
- `OptLen`：按理想比例压缩后的理论长度。
- `Start`：考虑固定时间、执行开始后的实际计算开始。
- `ActLen`：当前实际分配长度。
- `Delay`：`Start - OptStart`。
- `Percent`：`ActLen / OptLen`。
- `OptShift`：`OptStart - Start`，可作为高级列默认隐藏。

### 3. Reset

将计划恢复为理论最优状态。

建议行为：

- 清除所有执行字段：
  - actualStart
  - actualEnd
  - skipped
  - terminatedAt
- 可选择是否清除 fixedStart。
- 重新计算所有活动。

### 4. Adjust from now / Adjust from selected

用于执行中的局部重算：

- `Adjust from now`：保留当前时间之前的执行结果，从当前时间重新压缩后续活动。
- `Adjust from selected`：保留所选活动之前的结果，从所选活动开始重算。
- 需要清楚显示哪些活动会被改动。

## P1：统计与复盘

### 1. 活动关键词统计

复刻文档中的 Totals 思路。

建议规则：

- 默认用活动标题第一个词或冒号前缀作为统计关键词。
  - `SuperMemo：核心复习` -> `SuperMemo`
  - `Work: Creativity` -> `Work`
- 允许用户手动设置统计关键词。
- 每日显示关键词总时长。
- 历史页面显示周/月汇总。

### 2. 手动统计标记

支持文档中的 `++Keyword 20` 风格，但不要强制用户写特殊文本。

建议以结构化 UI 实现：

- 每个活动可添加多个统计条目。
- 统计条目字段：
  - keyword
  - minutes
  - note
- 示例：
  - 活动：Sports
  - 统计条目：Jogging 20、Gym 15、Sauna 20

### 3. 延误趋势

帮助用户发现长期高估/低估。

建议加入：

- 最近 7 天延误排行。
- 最近 30 天被压缩最多的活动。
- 总是超时的活动。
- 总是被挤掉的活动。
- 建议调整值：
  - 使用中位数而不是平均值，减少异常日影响。

### 4. 完成质量标记

时间长度不等于质量。v2 可加入轻量标记：

- 完成质量：好 / 一般 / 差。
- 中断次数。
- 备注。
- 这些字段只用于复盘，不参与算法。

### 1. 模板复制与版本化

- 复制模板。
- 重命名模板。
- 删除模板前二次确认。
- 标记默认模板。
- 模板保存版本：
  - 每次 Adjust 生成一个历史版本。
  - 可回滚到之前版本。

### 2. 模板分组

适合多场景：

- 工作日
- 周末
- 学习块
- 深度工作
- 旅行日
- 低能量日

### 3. 从模板快速创建时间块

创建今日副本时可修改：

- 开始时间。
- 总时长。
- 是否保留 fixedStart。
- 是否保留 rigid。
- 是否跳过某些活动。

## P1：数据模型增强

建议新增字段：

```ts
interface Activity {
  id: string;
  title: string;
  desiredMinutes: number;
  fixedStart?: string;
  actualStart?: string;
  actualEnd?: string;
  rigid?: boolean;
  skipped?: boolean;
  statsKey?: string;
  notes?: string;
  quality?: "good" | "ok" | "bad";
  interruptions?: number;
}

interface PlanDocument {
  id: string;
  name: string;
  mode: "template" | "execution" | "history";
  startTime: string;
  durationMinutes: number;
  activities: Activity[];
  sourceTemplateId?: string;
  terminatedAt?: string;
  createdAt: string;
  updatedAt: string;
  archivedAt?: string;
}
```

本地存储建议：

- 增加 `schemaVersion`。
- 写 migration 函数。
- 保存前做基本数据校验。
- 解析失败时不要覆盖旧数据，给出恢复入口。

## P2：算法增强

### 1. 最小时长保护

避免活动被压缩到不可执行。

字段：

- `minMinutes?: number`

行为：

- 先满足所有 minMinutes。
- 剩余时间再按比例分配。
- 如果总 minMinutes 超过计划总长，显示冲突。

### 2. 最大时长限制

避免某些活动因剩余时间过多被拉长。

字段：

- `maxMinutes?: number`

适用场景：

- 邮件最多 30 分钟。
- 休息最多 20 分钟。

### 3. 优先级压缩

当时间不够时，不一定所有活动等比例压缩。

字段：

- `priority: 1 | 2 | 3 | 4 | 5`

行为：

- 高优先级活动保留更多时间。
- 低优先级活动更容易被压缩。
- 需要在 UI 中解释清楚，避免用户误解。

### 4. 约束冲突解释器

当固定时间、刚性时段、最小时长互相冲突时，给出人能读懂的错误：

- 哪些活动导致冲突。
- 超出多少分钟。
- 建议解除哪个 fixedStart / rigid / minMinutes。

## P3：可选高级功能

### 1. Random / By day 活动参数

复刻 SuperMemo 文档中的活动参数：

- 随机选择：
  - `{#R: 16 Read NS | 10 Read cnn.com | 13 Read SciAm}`
- 按星期选择：
  - `{#D: Mon activity | Tue activity | ...}`

建议不要直接暴露原始语法，优先做结构化编辑器：

- Choice method：Random / By weekday。
- 候选活动列表。
- 每个候选活动有标题和时长。
- 创建今日副本时解析成当天具体活动。

### 2. PWA

- 离线安装。
- App 图标。
- Service Worker 缓存静态资源。
- 本地数据仍保存在浏览器。

### 3. 云同步

只有当本地版本稳定后再考虑。

可能方案：

- Supabase Auth + Postgres。
- 端到端加密本地数据后同步。
- 多设备冲突解决。

v2 不建议直接做云同步，除非明确需要多设备。

当前 v1 为了快速交付，主界面集中在 `App.tsx`。v2 应拆分：

- `components/AppShell.tsx`
- `components/TemplateSelector.tsx`
- `components/PlanControls.tsx`

- `components/AlarmModal.tsx`

建议拆分：

- `schedule/computeSchedule.ts`
- `schedule/delayAnalysis.ts`
- `schedule/execution.ts`
- `schedule/adjust.ts`
- `schedule/constraints.ts`
- `schedule/importExport.ts`

新增测试：

- Rigid 活动压缩。

## v2 不建议优先做的内容

- 云端账户系统。
- 多人协作。
- 复杂项目管理功能。
- 甘特图。
- 任务依赖。
- AI 自动排程。
- 与 Outlook / Google Calendar 双向同步。

原因：这些功能会把产品从 SuperMemo Plan 风格的“比例时间管理工具”拉向通用项目管理或日历工具，容易稀释核心体验。
